# 玩家行动设计（小型验证版）

当前行动系统采用 DND5E 风格的能力检定作为底层规则，并为前端加入显式投骰表现。

## 判定与时间
- DC 与时间消耗由 AI 基于行动描述生成，要求结果可解释、可回放。
- 当前主要对 `attack`、`check`、`item_use` 三类行为执行检定。
- 基础规则为 `1d20 + 相关能力修正` 对比 `DC`。
- 天然 `1` 视为大失败，天然 `20` 视为大成功。
- 检定失败仍然推进时间，并可能附带资源或关系层面的负面后果。
- 时间仍然是关键资源；行动结束后 UI 继续用非阻塞提示展示耗时增加。

## 能力映射
- 由 AI 判定本次行动使用的能力项。
- 当前实现使用 DND5E 能力修正值参与检定结算。
- 玩家与 NPC 使用同一套能力检定链路。

## 检定表现
- 调用检定 API 前，前端必须先弹出全屏锁定投骰窗口。
- 投骰窗口出现后，玩家无法继续进行聊天、地图或其他操作。
- 窗口中央展示一个 `d20` 骰子，并保持明显的待触发状态。
- 玩家点击空白区域后开始掷骰动画。
- 动画节奏参考 Google 骰子工具：先明显滚动，再停在最终面。
- 动画结束后先显示本次骰点，再显示 `骰点 + 修正值 = 总分` 与 `DC` 的比较。
- 比较结果必须明确展示：
  - 是否成功
  - 是否天然 1 / 天然 20
  - 使用的能力与修正值
  - 最终叙事结论
- 玩家点击“继续”后，才关闭检定窗口并回到原界面。

## 一致性要求
- 前端动画展示的骰点必须等于后端最终结算使用的骰点。
- 不允许出现“动画掷出 17，但服务端按 6 结算”的不一致情况。
- 因此前端在动画结束后，需要将最终 `d20` 点数作为结构化字段传给后端。

## 装备与物品
- 装备/卸下武器与护甲使用固定时间消耗。
- 使用物品时，由 AI 判定是否需要检定与所需时间。
- 若需要检定，则仍然进入统一的投骰表现流程。

## 惩罚与后果
- 检定失败会修改角色当前资源或关系状态，并返回叙事后果。
- 大失败可以直接触发更严重的负面叙事。
- 大成功可以返回更高质量的结果描述，但仍需遵守当前系统资源规则。

## 调试与验证
- 调试面板中的“行为检定”入口也必须走同一套投骰流程。
- 物品使用触发的检定与调试面板触发的检定，视觉反馈、锁定逻辑、结算展示必须一致。
