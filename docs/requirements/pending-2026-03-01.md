## 1. 当前基线结论
- 已有基础能力：存档分片、地图/区块/时钟、角色池、NPC 单聊、行动检定、CSV prompt 表、阻塞遮罩弹窗基础样式。
- 当前缺口：仓库尚无命运线、任务、遭遇三套正式数据结构，无对应 API、前端弹窗、日志分类、调试入口与回归测试。
- 约束已明确：
  - 所有相关弹窗都必须阻塞聊天。
  - 命运任务为 `accept_only`。
  - 普通任务为 `accept/reject`，且操作结果要写入聊天叙事。
  - 弹窗优先级为 `命运任务/普通任务 > 遭遇`。
  - 遭遇全文必须能在日志面板查看。
  - 遭遇体验偏好存本地 CSV，用于全局 prompt。

## 3. P1 行为闭环

### 3.1 任务闭环
- [ ] 普通任务发放时进入 `pending_offer`，支持接受/拒绝。
- [ ] 普通任务接受/拒绝后，主聊天区输出叙事反馈。
- [ ] 命运任务发放时进入 `pending_offer`，只允许接受。
- [ ] 角色卡增加任务列表与任务日志区域。
- [ ] 角色卡支持“确认当前任务/设为追踪任务”。

### 3.2 遭遇闭环
- [ ] 玩家移动后支持遭遇检测。
- [ ] 玩家对话后支持遭遇检测。
- [ ] 遭遇生成时必须带上当前区域、当前任务和玩家全局偏好。
- [ ] 遭遇动作结算后把结果写入主聊天区。
- [ ] 遭遇内容与结算全文写入日志面板可查看的数据。

### 3.3 命运闭环
- [ ] 命运线阶段数据支持多阶段顺序推进。
- [ ] 每个阶段绑定一个命运任务。
- [ ] 命运任务完成后，对应命运阶段推进为完成。
- [ ] 支持手动执行“检查命运阶段条件”。

## 4. P2 自动化与深化

### 4.1 自动触发
- [ ] 命运阶段支持按天数/NPC/物品/遭遇/任务完成自动判定触发。
- [ ] 命运阶段满足条件时自动发放命运任务。
- [ ] 任务完成后自动触发依赖它的遭遇或下游命运阶段。

### 4.2 更强的语义联动
- [ ] 遭遇根据当前任务目标和命运阶段语义生成，而不是仅基于随机模板。
- [ ] 任务完成判定同时参考地图状态、角色关系、遭遇历史和最近聊天记录。
- [ ] 命运线生成结果可调用任务/遭遇生成能力，形成完整主线链路。

### 4.3 稳定性与可维护性
- [ ] 为三套系统补齐后端单元测试。
- [ ] 为关键前端弹窗流程补齐交互测试。
- [ ] 更新模块 README 和总技术索引，保证文档入口完整。

## 5. 验收标准
- [ ] 新建或读取旧存档后，三套新状态都能稳定补齐。
- [ ] 命运/任务/遭遇在 UI 中按优先级正确阻塞显示。
- [ ] 普通任务接受/拒绝、命运任务接受、遭遇动作都能写入聊天与日志。
- [ ] 玩家能在角色卡中看到任务列表和任务日志。
- [ ] Debug 面板能独立触发命运线、任务和强制遭遇。
- [ ] 日志面板能查看完整遭遇内容。
